Flow hoạt động của CACHE (filter)

1. MISS Cache:
    Request
      => Spring Cache Proxy intercept
          => cache NOT FOUND (MISS)
          => gọi method thật
              => AOP (CacheHeaderAspect) chạy:
                    set:
                      X_CACHE = MISS
                      X_CACHE_NAME = publicCert   (lấy từ @Cacheable)
                      X_CACHE_KEY = cert:CERT_123 (evaluate từ SpEL key)
              => method business chạy (DB, RSA, ...)
              => kết quả được lưu vào cache
      => Filter (CacheHeaderFilter) chạy cuối:
            đọc attribute từ request
            set header vào response

    Response:
      X-Cache: MISS
      X-Cache-Name: publicCert
      X-Cache-Key: cert:CERT_123

2. HIT Cache:
    Request
      => Spring Cache Proxy intercept
          => cache FOUND (HIT)
          => return data ngay (KHÔNG gọi method)
          => AOP KHÔNG chạy
          => không có attribute nào được set

      => Filter (CacheHeaderFilter):
            không thấy X_CACHE => default = HIT
            không có cacheName / key

    Response:
      X-Cache: HIT

Insight: MISS => đi qua method => AOP có dữ liệu => đầy đủ header
         HIT  => không đi qua method => AOP không chạy => chỉ biết HIT

Pipeline: Client
            ↓
          Filter (MdcFilter)
            ↓
          Spring Cache (Proxy)
            ↓
          AOP (chỉ khi MISS)
            ↓
          Service Method
            ↓
          Filter (CacheHeaderFilter)
            ↓
          Response

----------------------------------DETAIL - E EXPERIENCE LAYER
1. TRIGGER LAYER — MilestoneScheduler
    + Bản chất: System-driven (không cần user action)
    + Cơ chế hoạt động: @Scheduled (cron hoặc fixedRate) => tự động chạy theo thời gian
    + Flow: MilestoneScheduler => gọi MilestoneService.processRelationshipMilestones()
    + Vai trò:+ Trigger business event (milestone reached)
            + Không xử lý retry
            + Không giữ state

2. BUSINESS LAYER — MilestoneService
    + Bản chất: Core business logic của Experience Layer
    + Flow: [1] Load tất cả CertificateSession có status = COMPLETED

                     [2] Tính:
                         daysSinceRelationshipStart = today - loveStartDate

                     [3] So sánh với milestone config (0, 30, 100, 365...)

                     [4] Lọc ra candidate sessions (chỉ những session hit milestone hôm nay)

                     [5] Query batch MilestoneLog:
                         => tránh N+1
                         => kiểm tra đã gửi chưa (idempotency)

                     [6] Nếu chưa gửi:
                         => trigger EmailService (@Async)

                     [7] Ghi log:
                         => lưu MilestoneLog (đảm bảo không gửi lại)
    + Vai trò: + Xác định event "milestone reached"
               + Đảm bảo idempotency (MilestoneLog)
               + Trigger notification (email)

3. ASYNC LAYER — EmailService
    + Bản chất: Non-blocking execution (tách khỏi scheduler thread)
    + Flow: MilestoneService
            => EmailService.sendMilestoneEmail() (@Async)

                TRY:
                    gửi email qua SMTP
                    => SUCCESS => DONE

                CATCH:
                    => lưu EmailJob vào DB (status = FAILED)
    + Vai trò: + Tách IO khỏi scheduler
               + Không làm chậm system
               + Đẩy failure xuống DB (EmailJob)
4. QUEUE LAYER (DB-based) — EmailJob
    + Bản chất: Database đóng vai trò Message Queue
    + State machine: PENDING => PROCESSING => SENT
                                     => FAILED => RETRY
                                     => DEAD
    + Vai trò: + Lưu job bị fail
               + Là nguồn dữ liệu cho retry
               + Đảm bảo durability (không mất job)

5. RETRY LAYER — EmailRetryScheduler
    + Bản chất: Worker (consumer) của DB queue
    + Flow: @Scheduled (mỗi 1 phút)
            [1] Claim jobs:
                UPDATE email_job
                SET status = PROCESSING, processing_at = NOW()
                WHERE status IN (FAILED, PENDING)
                  AND next_retry_at <= NOW()
                LIMIT N

            => đảm bảo: mỗi job chỉ được 1 instance xử lý

            [2] Load jobs có status = PROCESSING

            [3] Với mỗi job:
                TRY:
                    gửi email
                    => SUCCESS => status = SENT

                CATCH:
                    => tăng retryCount
                    => nếu < maxRetry => FAILED + nextRetryAt
                    => nếu >= maxRetry => DEAD
    + Vai trò: + Retry email bị fail
               + Xử lý queue
               + Áp dụng backoff retry
               + Đảm bảo không duplicate (DB locking)

6. RECOVERY LAYER — EmailRecoverScheduler
    + Bản chất: Self-healing mechanism
    + Flow: @Scheduled (mỗi 1 phút)

            [1] Tính threshold:
                now - timeout (vd: 5 phút)

            [2] Update:
                PROCESSING => FAILED
                nếu processing_at < threshold

            => nghĩa là:
                job bị kẹt quá lâu => reset để retry lại
    + Vai trò: + Detect job bị "kẹt" (do crash / timeout)
               + Reset để retry tiếp
               + Đảm bảo system không bị dead job
